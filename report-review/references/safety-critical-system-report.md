# 高风险自动系统报告标准

当报告要评估一个错误会直接影响下游用户、污染生成结果、导致错误准出或产生明显坏结果的系统时，读取本 reference。适用对象包括匹配、路由、定位、grounding、检测、质量闸门、自动替换等高风险决策模块。

## 核心问题

报告必须回答：

> 这个系统是否安全到足以作为决策闸门或自动执行层？当它不确定时会如何处理？

高风险系统不能只看平均准确率，必须控制错误曝光，并说明 fallback / reject / 降级行为。

## 必备证据

1. **系统角色**
   - 它是最终闸门、匹配层、路由层、自动替换层、fallback 层，还是诊断模块？
   - 如果它错了，下游会受到什么伤害？

2. **安全优先指标**
   - 误放行、误匹配、坏结果曝光、有害替换、错误路由、严重用户可见错误。

3. **收益指标**
   - 召回、覆盖、匹配率、自动化率、节省人工、质量提升。

4. **覆盖 / fallback 指标**
   - 系统不确定时做什么？
   - 拒绝、fallback、降级、转人工、保留原结果或跳过。

5. **风险分桶**
   - 先定义风险桶，再解释结果。
   - 包含普通 case、模糊 case、重复/多义 case、长尾 case、噪声数据、高风险 case。

6. **阈值**
   - 安全阈值。
   - 收益阈值。
   - 覆盖阈值。
   - 分桶最低要求。

7. **失败策略**
   - 某个桶过不了时怎么办？
   - 当前版本明确不支持什么？

## 常见问题

| 问题 | 判断信号 | 纠偏方向 |
| --- | --- | --- |
| 平均值掩盖危险失败 | 总准确率高，但存在严重错误输出 | 补安全指标和严重 case 复核 |
| 用不安全换收益 | 覆盖或自动化提升，但误放行上升 | 安全指标优先 |
| 没有 fallback 策略 | 系统不确定时仍然执行动作 | 补拒绝 / fallback / 降级规则 |
| patch 式验证 | 只针对看过的 bad case 加规则 | 建固定回归集和分桶测试 |
| 低估系统角色 | 核心质量闸门被当成普通辅助模块 | 提升到准出闸门标准 |

## 推荐结构

1. 系统角色与下游风险。
2. 准出 / 闸门标准。
3. 样本口径与风险分桶。
4. 主安全指标。
5. 收益与覆盖指标。
6. fallback / reject / 降级行为。
7. 分桶失败与不支持范围。
8. 是否可作为正式闸门、灰度闸门、辅助模块或诊断工具。
9. 回归集与监控计划。

## 示例

通用规则：

> 如果一个模块决定结果是否进入下游，就必须按“闸门”而不是“辅助工具”评估。它的误放行和严重错误行为，比平均提升更重要。

历史示例：

在一个高风险自动决策工作流中，一个决定是否替换中间结果的匹配层，需要评估错误输出率、覆盖率、回退率、回退正确率、歧义处理和拒绝策略。这里真正可复用的标准不是具体字段格式，而是任何会暴露有害错误输出的自动决策层都必须按安全闸门审查。
