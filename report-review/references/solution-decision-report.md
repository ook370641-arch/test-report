# 方案决策报告标准

当报告要评估一个新方案、模块、prompt、模型、规则、流程或策略是否值得采用、灰度、继续投入或拒绝时，读取本 reference。

## 核心问题

报告必须回答：

> 相比清晰的基线，新方案是否在当前风险、成本和边界条件下创造了足够的净价值，因此值得采用？

这类报告不是只证明“方案能跑”，而是证明“方案值得用”。

## 必备证据

1. **基线与对照**
   - 当前基线是什么？
   - 新方案是什么？
   - 如果可能，是否有去掉关键新增模块后的对照版本？
   - 当前隔离的关键变量是什么？

2. **决策层级**
   - 能跑：流程可以执行并产生结果。
   - 有效：局部指标或局部 case 有改善。
   - 值得采用：净收益大于风险和成本。
   - 可准出：同时达到绝对阈值、安全边界和失败边界要求。

3. **指标层级**
   - 主指标：直接回答是否值得采用。
   - 安全指标：防止有害结果被放行。
   - 收益指标：衡量正向价值。
   - 覆盖 / 成本指标：衡量覆盖损失、延迟、维护复杂度、人工成本。
   - 边界指标：说明方案在哪些场景失败。

4. **绝对标准**
   - 相对变好不等于可以准出。
   - 报告必须说明新方案是否达到最低可接受标准。

5. **最终决策**
   - 采用 / 拒绝 / 灰度 / 继续验证 / 仅作诊断。
   - 同时说明什么证据会改变当前决策。

## 常见问题

| 问题 | 判断信号 | 纠偏方向 |
| --- | --- | --- |
| 只证明可行，没证明价值 | 报告说方案能跑，但没说明值不值得采用 | 补基线、成本、风险和净收益 |
| 缺少控制变量 | 提升可能来自多个变化 | 补基线、消融或关键变量隔离对照 |
| 只有相对结论 | “B 比 A 好”被写成可准出 | 补绝对阈值和安全边界 |
| 收益掩盖安全成本 | 收益指标提升，但错误曝光或误放行未被讨论 | 安全指标优先于收益指标 |
| 平均值掩盖边界 | 均值变好，但困难场景未拆开 | 补风险桶和失败桶 |

## 推荐结构

1. 主问题与当前决策。
2. 基线、新方案和关键变量。
3. 样本 / 事实口径。
4. 成功标准与准出标准。
5. 主结果。
6. 安全、收益、覆盖、成本指标。
7. 边界与失败分桶。
8. 净收益解释。
9. 最终决策与下一步验证。

## 示例

通用规则：

> 不要写“新模块改善了一些 case，所以应该采用”。应写“相比基线，完整方案提升了主指标，未牺牲安全指标，达到准出阈值，成本可接受，因此可以采用 / 灰度 / 拒绝”。

历史示例：

在评估一个新增强模块是否值得采用时，不能只说“模块 A 让部分结果变好了”。应比较基线系统、引入部分改动的中间版本、完整新方案，并同时判断主指标提升、错误率、覆盖率、回退正确率和维护成本。
