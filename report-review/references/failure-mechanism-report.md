# 失败机制归因报告标准

当报告要分析 bad case、失败样本、边界样本、长尾失败、回归问题或意外退化时，读取本 reference。

## 核心问题

报告必须回答：

> 出现了什么失败模式，为什么发生，哪些其他解释已经被排除，以及这个机制对应什么策略？

bad case 是诊断证据，不会自动构成结论。

## 必备证据

1. **失败范围**
   - bad case 来自哪个样本池？
   - 是全量失败、分桶失败、A/B 差异样本，还是人工挑选样本？

2. **具象展示**
   - 必须让读者能直接检查失败现场。
   - 包含输入、输出、期望行为、实际行为、判定理由。

3. **模式提取**
   - 按共同机制分组，不只按表层标签分组。
   - 说明多个 case 的共同点。

4. **触发变量**
   - 哪些输入、上下文、数据条件、模型行为、规则、prompt 或流程变化可能触发失败？

5. **排除项**
   - 程序错误。
   - 字段映射或解析错误。
   - 源数据噪声。
   - 评测工具误判。
   - 判定标准歧义。

6. **归因强度**
   - 已验证机制。
   - 高置信观察。
   - 待验证假设。
   - 下一步验证策略。

7. **对应策略**
   - 修复、过滤、分级、降级、拒绝、灰度、监控或停止采用。

## A/B bad case 分析

当 bad case 来自 A/B 测试时，必须先使用以下对照单元，再做机制归因：

| 维度 | 必须说明 |
| --- | --- |
| A 输入 / 基线输入 | 原始信息、约束、样本口径 |
| B 输入 / 新方案输入 | 新增、删除或改变的信息、规则、字段、prompt 或流程 |
| A 输出 | 保留了什么、失败了什么、哪些能力仍然存在 |
| B 输出 | 改善了什么、退化了什么、缺失了什么、偏移了什么 |
| 人工判定与排除项 | 差异是否真实；是否排除程序、字段、数据和评测问题 |

## 常见问题

| 问题 | 判断信号 | 纠偏方向 |
| --- | --- | --- |
| case 堆放 | 很多截图或例子，但没有模式 | 按机制和触发变量分组 |
| 只用数量下结论 | 失败数量增加后直接断言原因 | 补输入/输出对照和排除项 |
| 把现象当机制 | “输出坍缩”等现象被当成解释 | 说明什么变了、为什么变 |
| 没有排除项 | 未排除程序、字段、源数据或评测误判 | 补排除清单 |
| 策略过泛 | 只写“优化 prompt”“补数据” | 每类机制绑定具体策略 |

## 推荐结构

1. 失败问题。
2. 失败样本口径。
3. 最具代表性的 case。
4. 失败模式表。
5. 触发变量分析。
6. 排除项清单。
7. 归因强度标注。
8. 策略与验证计划。

## 示例

通用规则：

> bad case 报告不能停在“这种失败更多了”。它必须说明是什么输入或条件导致失败，为什么其他解释较不可能，以及由此推出什么决策。

历史示例：

在一次方案 A/B 中，新增输入有时能提升整体表现，但也会让部分输出丢失关键中间步骤。真正可复用的标准不是某个字段名称，而是方法：比较基线输入/输出与新方案输入/输出，排除字段或评测错误，再判断新增输入是有效上下文、重复上下文，还是有害引导。
