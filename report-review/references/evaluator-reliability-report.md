# 评测工具可靠性报告标准

当报告要评估某个“评测尺子”是否可靠时，读取本 reference。评测尺子包括：质检 prompt、自动评分脚本、LLM-as-judge、规则判定器、人工 rubric，以及任何输出通过/不通过、好/坏、可用/不可用的系统。

## 核心问题

报告必须回答：

> 这个评测工具是否可靠到足以作为后续决策的测量依据？

如果评测工具本身不可靠，被评对象的结论必须降级。

## 必备证据

1. **评测工具角色**
   - 它判断什么？
   - 哪个决策依赖它的输出？
   - 它是准出闸门、诊断工具，还是临时筛查工具？

2. **可信基准**
   - 与什么标准对齐？
   - 人工复核、已知标签、源答案、源方法、专家 rubric 或人工盲评。

3. **错误类型**
   - 误杀 / 过严：好结果被拒绝。
   - 误放行 / 漏拦：坏结果被接受。
   - 边界漂移：同类边界 case 判断不稳定。
   - 标准错位：评测工具优化的目标与真实任务不一致。

4. **校准样本**
   - 必须包含通过样本和未通过样本。
   - 必须包含明显 case、模糊 case、边界 case。
   - 必须包含已知高风险 case。

5. **稳定性验证**
   - 对 LLM 或概率性评测工具，关键 case 需要重复评测。
   - P0/P1 或准出卡点 case 不应在重复运行中漂移。

6. **结论降级**
   - 如果评测工具尚未完全可靠，下游结果只能写成阶段性观察，不能写成最终决策。

## 常见问题

| 问题 | 判断信号 | 纠偏方向 |
| --- | --- | --- |
| 把复核动作当成可靠性证明 | “做了人工抽查”直接推出工具可靠 | 补样本范围、错误类型和残余风险 |
| 混淆工具与被评对象 | 一边承认尺子不稳，一边用它给方案下最终结论 | 先验证尺子，再限定被评对象结论 |
| 只看误杀，不看误放行 | 只讨论好结果被错杀 | 补漏拦、误放行和坏结果进入下游的风险 |
| 缺少稳定性验证 | LLM 只跑一次关键 case | 对高风险 case 重复评测或加人工校准 |
| 忽略 rubric 偏差 | 高分与真实人工体感不一致 | 补人工盲评或 rubric 对齐 |

## 推荐结构

1. 评测工具用途与决策角色。
2. 可信基准 / 人工 rubric。
3. 校准样本口径。
4. 错误类型与错误率。
5. 稳定性 / 重复评测结果。
6. 可靠范围与不可靠范围。
7. 能否作为准出闸门、灰度闸门、诊断工具，或暂不可用。
8. 哪些下游结论必须降级。

## 示例

通用规则：

> 先证明尺子可靠，再用尺子证明被评对象可靠。

历史示例：

在用一个自动评分规则比较两个方案时，发现同一批样本在两种方案下存在通过/不通过差异。报告必须先验证该评分规则是否存在误放行、误杀和边界漂移，再把通过率作为版本效果证据。
